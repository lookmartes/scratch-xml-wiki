Troubleshooting

Solutions to common errors and unexpected behaviour when using the Scratch XML Injector.

Reference

01Common Errors

Error / SymptomCauseFix
VM not found The injector couldn't locate the Scratch VM instance in the page. Do a hard reload (Ctrl+Shift+R) of the editor page, then run the script again. Make sure you're inside the project editor (after clicking "See inside"), not just the project page.
FixedAsciiString error in browser console Non-ASCII characters (smart quotes, emoji, accented letters) in your XML were passed to Scratch's asset validator. Click the ~ Strip Non-ASCII button in the panel before injecting, or write your XML in a plain text editor that doesn't auto-correct characters.
XML parse error Your XML is malformed — missing a closing tag, unclosed attribute, or invalid character. Validate your XML at xmlvalidation.com. The most common mistake is a missing </next> or </statement>.
Blocks injected but workspace is empty The injector targeted the wrong sprite, or a workspace/reload conflict occurred. Make sure the correct sprite is selected in the sprite list. Try the injection again — a second attempt often succeeds after the VM settles.
Project build fails silently An error occurred during the project rebuild that was caught internally. Open browser DevTools (F12 → Console tab) — the full error message is always logged there with an [SXI] prefix.
Sprites appear with no costume / blank white square The type attribute on <costume> is missing or misspelled. Check that type is exactly circle or rectangle (lowercase).
Panel doesn't appear Tampermonkey is disabled on this tab, or the script ran while the panel was already open (toggle closed). Click the Tampermonkey icon and verify it's enabled for this site. Check that the script is active in Dashboard.
"Not a <project> XML" error in Project mode You're in Project XML mode but pasted Blocks XML (<xml> root). Switch the mode toggle to ~ Blocks XML, or change the root element of your document to <project>.
Blocks appear but don't work when flag clicked A block opcode string is incorrect or an input type mismatches. Use the Export tab on a working block in Scratch to see its correct XML, then compare to your code.

02Non-ASCII Characters in Depth

Scratch's internal asset storage uses a validator called FixedAsciiString that rejects any string containing characters above Unicode codepoint 127. This includes:

Smart quotes " " Em dash — Ellipsis … Accents é ñ ü Emoji 🎮 Non-breaking space

These often appear when copying XML from a word processor, rich text editor, or web page that applies "smart" formatting.

✓
The injector automatically strips non-ASCII characters before each injection attempt. You can also click ~ Strip Non-ASCII manually. The yellow badge "! non-ASCII detected" appears when your pasted XML contains problem characters.

03Debugging Tips

Use the Export tab first

The single most powerful debugging technique: build the block you're trying to create manually in Scratch's block editor, then use the Export tab to see exactly what the injector-compatible XML looks like. Compare it character-by-character with your code.

Check the browser console

Open DevTools (F12) → Console. Every error the injector catches is logged with an [SXI v6.x] prefix. The full stack trace and error message will tell you exactly what went wrong.

Validate your XML

An XML validator will catch structural errors before you even try to inject. Paste your XML at xmlvalidation.com and fix any parse errors first.

Test with a minimal example

If a large block of XML fails, strip it down to the smallest possible working unit (just the hat block), then add pieces back one at a time until you find the broken section.
