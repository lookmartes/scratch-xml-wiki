Inject Modes

v7.0 introduces four distinct injection modes — each controls exactly what gets overwritten and what stays untouched.

v7.0Reference

Overview

The mode buttons appear at the top of the ~> Inject tab. The active mode is highlighted and changes both the label and the action button text/color.

ModeXML formatScopeSafe?
[#] New Project <project> Entire project — all sprites wiped Destructive
[+] Add to Project <project> Only named sprites in the XML Partial
~ Replace Blocks <xml> Active sprite blocks only Sprite blocks
[+] Append Blocks <xml> Adds to active sprite, deletes nothing Non-destructive

Mode 1[#] New Project

Completely wipes the current Scratch project and rebuilds it from the <project> XML you provide. Every sprite, every block, every costume from the old project is gone.

When to use it

Use when you're starting fresh — generating a whole game from scratch, or replacing a prototype with a new version you've designed outside Scratch.

What you need

A valid <project> XML document. See the Project XML guide for the full format.

⚠
Save first. This is irreversible. Use File → Save to your computer before running this mode.

Mode 2[+] Add to Project

Parses a <project> XML and merges its sprites into the running project. Existing sprites that aren't named in the XML are completely untouched.

Merge logic

1

Sprite not in project → created

A brand new sprite is added with the costumes and blocks defined in the XML. It appears at the top of the layer order.

2

Sprite name matches existing → updated

The existing sprite's blocks are replaced by the XML's blocks. If the XML includes <costume> elements, those replace the existing costumes too. Position (x/y), size, and other properties are updated.

3

Sprite not in XML → untouched

Any sprite in the project that isn't mentioned in the XML keeps all its blocks, costumes, variables, and sounds exactly as-is.

ℹ
Name matching is case-sensitive. "player" and "Player" are treated as different sprites.

Mode 3~ Replace Blocks

The classic v6.3 "Blocks XML" inject behaviour. Clears the active sprite's workspace and loads the new <xml> blocks. All other sprites are untouched.

When to use it

Use when you want to give a sprite a completely new set of scripts, replacing whatever was there before.

How it works

If the Blockly workspace is accessible, it calls workspace.clear() then Blockly.Xml.domToWorkspace(). If not (e.g. workspace not rendered), it patches the sprite's blocks directly in the project JSON and reloads.

Mode 4[+] Append Blocks

Adds new block stacks to the active sprite's workspace without clearing anything. Existing scripts are preserved exactly as-is; the new blocks appear alongside them.

When to use it

Use when you want to add extra scripts — a new event handler, a helper stack, a second when flag clicked — without replacing your existing work.

Tip: Use x/y to place new stacks

Because existing stacks are kept, new stacks may overlap visually. Use the x and y attributes on top-level <block> elements to position them away from existing scripts:

XML — Position new stack away from existing ones
<xml xmlns="https://developers.google.com/blockly/xml">
  <!-- x=600 puts it well to the right of existing scripts -->
  <block type="event_whenkeypressed" x="600" y="60">
    <field name="KEY_OPTION">space</field>
    <next>
      <block type="looks_say">
        <value name="MESSAGE">
          <shadow type="text">
            <field name="TEXT">Jump!</field>
          </shadow>
        </value>
      </block>
    </next>
  </block>
</xml>
